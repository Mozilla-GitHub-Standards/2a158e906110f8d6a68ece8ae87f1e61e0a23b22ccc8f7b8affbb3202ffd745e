Nachdem in Bugzilla hauptsächlich Englisch gesprochen wird ist zu empfehlen das selbst deutsche Tech Evangelism Bugs auf englisch eingegeben werden. Das hat den Vorteil, dass viel mehr Leute Ihr Problem verstehen bzw. nachvollziehen können und es so eventuell zu einer schnelleren Lösung kommt. Sollten Ihre Englischkenntnisse (Ihrer Meinung nach) nicht ausreichend sein, können Sie Bugs natürlich auch auf deutsch posten.

Was sind Tech Evangelism Bugs?

Tech Evangelism in Bugzilla ist ein komplett anderes Projekt, gegenüber den restlichen mozilla.org Projekten. Dies kommt daher das diese Bugs fundamentale Unterschiede zu sogenannten Produktbugs besitzen. Ein Produktbug ist entweder ein Problem oder eine Erweiterung zu einem bestimmten mozilla.org Projekt, während ein Technologie Evangelism Bug ein Bericht über das nicht funktionieren von Mozilla mit einer bestimmten Webseite ist und das Problem auch dort (auf der Website) liegt.

Der Unterschied könnte größer nicht sein. Mozilla.org hat die Kontrolle über das lösen von Problemen in mozilla.org Projekten, während es keinerlei Einfluß auf externe Webseiten oder Produkte hat.

Wie schreibe ich einen Tech Evangelism Bug?

  1. Mozilla Tech Evangelism wurde nicht geboren, um Ihr Beschwerdeservice zu sein. Bevor Sie einen Tech Evangelism Bug in Bugzilla anlegen, sollten Sie versucht haben mit dem Webseitenbetreiber in Verbindung zu treten und ihn über seine fehlende Unterstützung für Mozilla zu informieren.

  2. Seien Sie sicher, dass das Problem für das Sie einen Bug eröffnen wollen wirklich ein Tech Evangelism Problem ist. Sind Sie nicht sicher, so eröffnen Sie einen Bug über das am plausibel erscheinenste Produkt .

  3. Eröffnen Sie einen neuen Tech Evangelism Bug in Bugzilla.

  4. Wählen Sie die richtige Komponente , in Bezug auf die verwendete Sprache der Website.

  5. Wählen Sie ihr Computer- und Betriebssystem.

    Wenn Sie mit Sicherheit wissen das dieses Problem nicht Computer- oder Betriebssystem abhängig ist, setzten Sie Platform und Operating System auf All.

  6. Wählen Sie den Anfangszustand des Bugs

    Wählen Sie UNCONFIRMED wenn Sie meinen das sich die Mozilla Evangelism Community diese Website näher anschauen sollte. Verwenden Sie NEW wenn Sie wissen welche Probleme Mozilla mit dieser Website hat.

  7. Wählen Sie den Schweregrad.

    Verwenden Sie major, normal oder minor wenn Sie einen Tech Evangelism Bug ausfüllen. Die anderen Schweregrade haben für das Tech Evangelism Projekt keinerlei Auswirkungen. Entscheiden Sie selber wie schwerwiegend das aufgetretene Problem ist.

  8. Bug zuteilen

    Wenn Sie sich selber um die Kontaktierung des Webseitenbetreibers kümmern wollen, setzten Sie in das Assigned To Feld Ihre eigene Emailadresse ein. Ansonsten lassen Sie das entsprechende Feld leer und der Verantwortliche dieser Komponente wird automatisch eingesetzt.

  9. Geben Sie die URL ein

    Geben Sie die URL der Website ein, die Probleme unter Mozilla verursacht. Sollte eine spezielle Webseite der Homepage Probleme mit Mozilla haben, ist es leicht möglich das auch andere Webseiten dieser Homepage ähnliche Probleme machen. Schauen Sie ob auf anderen Webseiten dieser Homepage ähnliche Probleme auftreten und wenn ja dann geben Sie bitte im URL-Feld die Einstiegsadresse zu dieser Homepage an.

  10. Geben Sie eine Zusammenfassung ein

    Geben Sie den 'Hauptteil' der URL gefolgt von eiem '-' ein. Danach folgt eine kurze Beschreibung des Problems. Dabei müssen keineswegs technische Begriffe fallen.

    Wollen Sie zum Beispiel ein Problem auf http://www.foo.bar.com/help/support/ melden, dann wäre der 'Hauptteil' der URL bar.com. Dies ermöglicht es anderen Leuten nach dieser URL zu suchen.

    • bar.com - Displays differently than IE5
    • netscape.com - Does not allow Mozilla
    • fubar.co.uk - Does not support secure connections in Mozilla

    Die Zusammenfassung ist etwas das jeder normale Benutzer schreiben und verstehen sollte. Ausführliche technische Beschreibungen des Problems werden dann dem Whiteboard Status hinzugefügt, sobald der Bug jemanden zugewiesen wurde.

  11. Geben Sie die Beschreibung des Problems ein.

    Geben Sie eine detaillierte, begründete Beschreibung des Problems ein. Viele der Felder in Bugzilla sind erst verfügbar wenn der Bug gespeichert wurde. Es reicht also wenn Sie eine prägnante, begründete Beschreibung des Problems hier eingeben. Sie können später weitere Informationen hinzufügen, wenn der Bug erst einmal gespeichert ist.

    Geben Sie auch jegliche nützliche Information in Bezug auf den 'Erstkontakt' mit dem Webseitenbetreiber ein. Selbstverfasste Beschwerden von Kunden (bzw. Webseitenbesucher) sind immer effektiver als ein anonymes Mozilla Evangelism Email. Wenn Sie wollen das die betreffende Website Mozilla unterstützt, so beschweren Sie sich doch!

  12. Übermitteln Sie den Bug

    Wenn Sie den Bug nicht übermitteln, wird er auch nicht gespeichert.

  13. Editieren Sie den eben angelegten Bug

    Sollte die Seite nicht richtig angezeigt werden, fügen Sie dem Bug doch einen Screenshot hinzu. Stellen Sie sicher das Sie bei dem Anhang den richtigen MIME-Typ angeben. Bitte versuchen Sie die Größe des Bildes so klein wie möglich zu halten. Wir brauchen kein 1MB großes Bild. Reduzieren Sie die Farbtiefe und die Abmessungen des Bildes um die Dateigröße zu minimieren. Die Hauptsache ist, dass man auf diesem Bild das angegebene Problem erkennen kann.

    Sollte die Webseite ungültiges HTML oder JavaScript verwenden, speichern Sie die entsprechende Datei lokal ab und fügen Sie sie mit dem MIME-Type text/plain dem Bug hinzu. Dies ermöglicht es uns den Originalzustand der Webseite und alle weiteren Änderungen zu dokumentieren.

Product

Tech Evangelism

Alle Tech Evangelism Bugs werden in Bugzilla unter dem Product Tech Evangelism geführt.

Component

Sprache basierend

Tech Evangelism ist nach Sprachen in einzelne components getrennt. Wählen Sie die passende Sprache für die entsprechende Webseite.

Priority

P1

Große Website, großes Problem - Dies sind die Fälle deren man sich als erster annehmen sollte.

Priority

P2

Große Website, kleines Problem - Versuchen Sie sich auf diese Bugs genauso wie auf die P1-Bugs zu konzentrieren. Speziell wenn das Problem leicht behoben werden kann

Priority

P3

Kleine Website, großes Problem - Behandlen Sie diese Fälle je nachdem wieviel Zeit Sie haben. Sollte das Problem zur Dokumentation für weitere Fälle hilfreich sein, kann man dem Ganzen auch eine höhere Priorität einräumen.

Priority

P4

Kleine Website, kleines Problem - Sollte nichts anderes zu tun sein, kümmert man sich um diese Fälle.

Status

UNCONFIRMED

Ein Status von UNCONFIRMED bedeutet, dass dieser Bug von einem neuen Mitglied der Community angelegt wurde, das noch nicht über die nötigen Rechte verfügt, direkt einen NEW Bug anzulegen.

QA these as they come in according to the ideas outlined in this document.

Status

NEW

Neue Bugs wurden entweder bestätigt oder von jemanden angelegt der über automatische Bestätigungsrechte in Bugzilla verfügt.

Status

ASSIGNED

Ändern Sie den Bugstatus zu ASSIGNED nur wenn Sie die Seite komplett analysiert haben und den Evangelism-Brief abgeschickt haben. Wenn Sie eine Webseite zum ersten Mal analysieren, dann schauen Sie sich nicht nur die gemeldete Webseite an. Schauen Sie sich auch einige andere Seiten dieser Webpräsenz an, um festzustellen ob dieses Problem generell auftritt. Danach schicken Sie den Evangelism Brief mit der beanstandeten URL (eventuell anderen betroffenen URLs) und einer Beschreibung des Problems (eventuell einer Lösungsmöglichkeit) an den verantwortlichen dieser Website.

Aktualisieren Sie den Bug über Vermerke im Whiteboard, welches weiter unten näher erklärt wird und ergänzen Sie die Probleme der Webseite durch Kommentare (verwendet LAYER-Tags, LAYER-API, IE4 API, ...).

Vergessen Sie nie, dass wir den Zuständigen extra Arbeit aufhalsen. Seien Sie also höflich, professionell und respektieren Sie ihr Gegenüber. Wir wollen ja schließlich nicht das jemand auf uns sauer ist.

Target Milestones

Monat zur Nachverfolgung / Future

Verwenden Sie Target Milestones um zu bestimmen, wann ein Bug nachbearbeitet werden muss. Tech Evangelism Bugs werden nun monatlich kontrolliert. Setzten Sie nur Milestones, wenn Sie den Bug ASSIGNED haben und der Brief auf dem Weg ist.

Bei einem Webmaster der auf zwei Evangelism Briefe nicht reagiert hat und/oder nicht vorhat die entsprechenden Probleme auf dessen Homepage zu beseitigen, wird der Target Milestone auf FUTURE gesetzt.

Resolution

WORKSFORME

Problem ist (für mich) nicht nachvollziehbar

Resolution

DUPLICATE

Diese Website wurde bereits wegen eines ähnlichen Problems gemeldet.

Resolution

FIXED

Das Problem der Website wurde behoben.

Resolution

WONTFIX

Verwenden Sie auf keinen Fall WONTFIX in Tech Evangelism Bugs. Stattdessen markieren Sie Websites, die negativ antworten mit not-responsive im Whiteboard Status.

Resolution

INVALID

Normalerweise wird INVALID in Tech Evangelism Bugs nicht verwendet. Allerdings können nach Ermessen des Komponentenbesitzer, Bugs auch als INVALID markiert werden.

Domain - kurze und prägnante Beschreibung des Problems.

netscape.com - mail uses document.layers

Status Whiteboard

not-responsive

Verwenden Sie not-responsive wenn der Webseitenbetreiber keinerlei Änderungen vornehmen will.

Status Whiteboard

technote-needed

Wenn Sie meinen das dieses Problem so häufig ist das es näher dokumentiert gehört, so fügen Sie dem Whiteboard Status ein technote-needed hinzu. Sobald der Bug zur genüge dokumentiert ist entfernen Sie den Eintrag technote-needed wieder.

Status Whiteboard

plugin

Verwenden Sie plugin im Whiteboard Status falls es notwendig ist den Hersteller eines Plugins zu kontaktieren.

Status Whiteboard

author

Verwenden Sie author im Whiteboard Status falls es notwendig ist den Hersteller bestimmter Software zu kontaktieren.

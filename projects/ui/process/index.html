Proposal for Mozilla User Experience Design Process     |     Draft

Initial Author: German Bauer, Netscape User Experience Group. Last updated 12/19/2000

This is a draft. Do not bounce FunBall(TM) on hard surfaces. May implode on impact.
This is where your feedback comes in. Please post all feedback to the
netscape.public.mozilla.ui newsgroup.

Goals

  • Improve Involvement of Mozilla contributors in Design Process
  • Increase avenues for feedback and influencing the project
  • Improve efficiency of decision making and design process
  • Allowing to serve different design needs of involved contributors

Design Process Specifics

A. Feedback
top ^

A.1 Bug system

In the past the bug system has not provided an effective means of dealing with input from the Mozilla community in terms of ideas for enhancement, reporting usability deficiencies, or simply stating design preferences.

Goals

  • Distribute the load of evaluating and confirming bugs
  • Get bugs to the correct owner more quickly
  • Encourage use of bug systems by making it easy to file meaningful bugs
  • Use newsgroups for discussions/opinions, use bugs when UE defects have been identified

Implementation Ideas

Each of the main components should have a UE feedback or front-end class of bugs that can be dealt with from within the component's UE owners

We also should also have a general UE component that deals with cross-applications and UE framework issues, such as overall menu framework, overall application window layout etc.

The bug system is NOT a place for design feedback, that's what newsgroups are for.

A.2 Mozilla User Interface Leads

The  Mozilla community selects a UI Lead within each component to act as the access point for feedback on UE issues. This will help ensuring that ideas, feedback and contributions from the Mozilla community get proper and timely responses.

B. Issues Meetings
top ^

UE Issues meetings are to be called by the module owner or UI leads (Mozilla or Netscape). The person calling the meeting is also responsible for collecting the agenda items before the meeting, distributing it to the meeting attendees, and running the meeting.

Goals

  • Facilitate working out design issues.

Implementation Ideas

Meetings are called as needed and are open to the UI lead from the Mozilla community. The UI lead is responsible for bringing  Mozilla specific issues to the table. Meetings notes will be posted to the component specific UE page.

C. Decision Making
top ^

Decisions will be made during issues meetings based on proposals for UE designs. If a discussion in an issues meeting does not yield a result that is agreeable to the parties involved in the meeting, a reasonable time frame will be set to allow for more discussion before the issue will get decided at a subsequent UE issues meeting.

Implementation Ideas

The final decision on a particular design is made by group consensus. Decisions may be reviewed only if the component owner and the UE owner for these components agree to reopen the issue.

Those contributing to the implementation of a decided UE design agree to adhere to the decided specification as close as possible. Should issues arise that make an implementation to spec impossible or difficult, the issue needs to be brought back to the module owner or UI lead to be re-discussed in an issues meeting.

D. UI Specifications
top ^

Specs act as a means to record and communicate the current state of a User Interface Design of a product feature using short descriptions of a goals, target users and specification details as well as illustrations as needed. These specs are accessible to anybody interested with an Internet connection.

UE Specification start out as proposals and serve as a basis for decision making in UE related issues meetings. Later, UE design specifications reflect the latest state of decision making on the UE design of a product feature.

Goals

  • Make UE specifications openly accessible to interested parties
  • Serve as proposals for UE designs
  • Reflect decisions made on UE issues

Implementation Ideas

Specifications of the common type open source components (formerly referred to by Netscape as 'Communicator') will be accessible at http://mozilla.org/projects/ui/communicator/. These will include: Browser, Mail, Editor, XP Toolkit and misc. sub components such as sidebar, bookmarks, history, FTP etc. Specifications can be located with their components, but will be linked to from this central UE Specifications page.

Other components can post their UE specifications in their own sub directories under http://mozilla.org/projects/ui/

Specifications can be changed only by the owner of the specification. Posting of alternate specs is encouraged to serve as a basis for UE discussions. Announcements of Posting, Discussions and Feedback should occur in the appropriate component newsgroups.

E. Different design needs
top ^

We currently have commercial builds as well as Mozilla builds, both serve different goals and a different customer sets. With more companies and organizations coming into the Mozilla world as contributors and subscribers we will have an even more diverse set of goals and target customers

UE design in particular is highly customer centric. Conflicting goals will arise for a number of features in the User Experience of different products and we have to allow for that, so that each involved party can properly target their User Experience to their customers.

On the other hand from a developers perspective, dividing the product into two distinct products is described as painful at best. It will make cross-pollination of technical advances more difficult.

Goals

  • Allow for customer centric implementation of User Experience for each contributor/ party
  • Minimize pain for developers by keeping forking of User Experience parts to a minimum to allow efficient sharing of technologies and resources

Implementation Ideas

Smart ideas will go here

Other Ideas

Allow a moderated area at http://mozilla.org/projects/ui/ideas/ that can house future project ideas related to user experience. This area would be moderated by the UI lead for the Mozilla community

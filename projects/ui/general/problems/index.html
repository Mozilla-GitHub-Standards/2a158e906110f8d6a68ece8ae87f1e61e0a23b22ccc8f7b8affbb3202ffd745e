Common problems in Mozilla interface design

Matthew Thomas

This document describes some of the more common mistakes in interface design for Mozilla. Some of these problems are specific issues with Mozilla; others are general problems found in all open source projects.

Too much interface

The problem

The single most effective method for improving the learnability and usability of an interface is to keep it simple. The simpler menus, windows, and dialogs are, the quicker it is for new users to become familiar with them, and the easier it is for expert users to concentrate on their task rather than finding their way through the interface.

However, programmers who have spent weeks or months implementing or maintaining a particular feature often vastly overestimate the amount which ordinary users know about it, or the degree to which users care about it. This leads the programmers to make interfaces which are too complicated, which use unnecessarily technical language, or which too closely reflect the underlying architecture rather than the user’s task. This problem is worse in open source projects such as Mozilla, where programmers may be designing for themselves rather than for the general public.

The solution

Take a step back and think about how much time users are likely to want to spend using your feature (and learning how to use it in the first place). Watch customers or family members using the feature in Mozilla, and the equivalent feature in competing applications. And simplify, simplify, simplify.

Repetition

The problem

If there are two methods of doing something, and the different methods are similarly accessible, users will usually waste more time deciding which method is faster (and possibly wondering if there is any difference in result between them) than they would have wasted by choosing the slower of the two methods. For example, having a Close button in a window for which a window manager already offers its own close button, causes a user to waste time hesitating between the two buttons.

The solution

Repetition is occasionally unavoidable, but usually it is a good predictor of an interface problem. The solution may be as simple as rearranging controls so that a single label applies to both of them; or it may involve fundamental code-level changes which will take a lot of effort. Ask your friendly local interface designer if you’re not sure what to do.

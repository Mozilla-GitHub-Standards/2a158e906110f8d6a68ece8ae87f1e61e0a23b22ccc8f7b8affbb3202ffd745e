External Routes

Provide the traditional OS ways to access components as separate apps.

  • Windows Quicklaunch Bar- Offer Shortcuts to Nav, Mail, and IM
  • Windows Start Menu - during install, offer user to set Navigator/Mail as default Browser/Mail app - place shortcuts in Start Menu
  • Desktop Shortcuts - Nav, Mail, IM
  • Systray Icon - provide shortcut to Nav, Mail, and IM, when quicklaunch is enabled
  • Use *Installation Courtesy* - allow user to opt out of desktop shortcut placement, and OS default browser/mail setting.
  • [Future] - Mac could use Application Bar for quicklaunch access and Component Shortcuts.
  • Most all of these objectives were agreeable and actionable






Internal Routes

Improving internal circulation within the application. Options 2 and 6 were agreeable paths to exploration.

  • Change Menu name and location - dependent on Menu Reorg work: from "tasks" to "windows" and add "tools"); see Prelminary Menu Reorg Proposal "B"
  • Add icons to accompany components (under the newly proposed "tools" menu)
  1. "Netscape Central" - a central location that concisely routes users to their desired task - primarily focused on a brand new user. This could be manifested in 3 ways:
      • A window - PROS: Familiar. Similar to PP, or Word. CONS: New window time is slow; feels like too many popups thanks to Netscape.com.
      • A browser Tab - PROS: Faster than a new window CONS: slower than what we currently have
      • A Sidebar Tab - PROS: Fast. CONS: Not enough room to present the icons and information.
  2. *"My Sidebar" area - PROS: Good relocation spot. CONS: Only available when sidebar is open
  3. Relocate a minimal Component Bar (Mail, or Mail and IM) to the area to the left of the Personal Toolbar - Dependent on implementing a column heading-like widget which, "drag and snaps" to varying degrees of compactness.
      • suggest removing the "navigator" component icon since it's redundant when you're already in Navigator
      • suggest removing the "address book" component icon from Navigator, because it is recognized and accessed as a Mail feature.
      • suggest removing the "composer/editor" icon, because editor is extended function of Navigator (operates on the same data type. doesn't receive the same frequency of use as the other components. can be accessed from the File Menu, as "Edit Page", which fits the Web Page Navigating/editing construct of our users.
      • Question: how badly does NIM need a component launcher icon? IM is the ubiquitous component which lives within sidebar. What data suggests that users currently use the product as a standalone launched from within the browser. Do the External Routes described above further justify this proposition?
      • How does this apply to Mail and Composer? Do we simply rely on Menus or External Routes to get back in
  4. "The 4.x way" - Not Mach5able
  5. "Sidebar Gutter" - Provide a vertically arranged set of component launchers, with Sidebar Tab status, along a new toolbar space which runs along the sidebar splitter. - Uses more horizontal space rather than more expensive vertical space.
  6. *Drawer - Not MachVable
  7. Start Button - Hides the components underneath a new special menu invoked from the personal toolbar.

Introduction

Tabs will represent one of the latest innovations in browsing in Mach V Netscape and beyond. The ultimate goal for the design of page tabs is that they pose no threat to the browsing practice of any user, and should provide seamless integrated user experience with windowed browsing, or any other browsing practices.


Competitive Tab Implementations

Opera, Net Captor, Galleon

Tab Interaction Design

Tabs should serve two primary functions while browsing

  1. A way to organize sites within a window, which facilitate common browsing scenarios (information traversal):
    • Drilling (i.e. breadcrumbing)
    • Perforation (i.e. search page index)
    • Comparison Shopping - the explicit organization of tabs into windows based on user attributed associations
    • "Coffee & Donuts" - routines (i.e. home set)
    • Windowless Browsing - (i.e. fullscreen mode) - a primary use scenario for laptops or space constrained screens.
  2. A flexible way to store frequently re-visited sites - sidebar storage mechanism

Feature Rollout

  • Tab Object Manipulation and Dragability implementation (http://bugzilla.mozilla.org/show_bug.cgi?id=126299):
    1. Tab focusing and highlighting
    2. Multiple tab selection
    3. Single and Multiple tab deletion (using the close button)
    4. Drag proxy to replace (to content area or a tab surface)
    5. Drag single and multiple proxy to Tab Bar (from bookmarks manager/sidebar)
    6. Insert single and multiple proxy between tabs (to tab bar with visual indication)
    7. Insert a Tab Group between existing tabs on tab bar
    8. Drag a Tab Group to conent area (insert and append after existing position in Tab Bar)
    9. Drag single and multiple tabs to bookmarks manager/sidebar
    10. Drag single and multiple tabs to personal toolbar (same as proxy to personal toolbar dragging)
    11. Drag a Tab Group to Tab Bar (from bookmark sidebar/manager)
    12. Drag a Tab Group to an empty window
    13. Drag arrange tabs within a window
  • Keyboard Navigation and Accessibility - how to integrate keyboard control of tabs so to not interfere with window convention. Tab commands should coexist peacefully with window. Never segregate tab users and window users. Don't kill the window user within.
    1. Ctrl-shift-tab - is focus jump right now
    2. Closing Current Tab: Ctrl-shift-w or Focus Shift-Delete / ctrl-w closes window
    3. Opens in new tab: Ctrl-shift-click open in new tab / ctrl-click opens in new windw
    4. Navigating tabs - Insert tabs in the focus order. Allow ctrl-tab to do focus jump from url, to tab bar, to sidebar, to page. Navigating tabs using arrow keys - left and right to move to focus, enter, up and down to scroll.
  • Context Menus
    • [Convert to Tab Group] --> found in PT, Bookmark Manager, Bookmark Sidebar, contexts
  • File Menus

Open Questions

  • What is the feasibility of Multiple Groupings within one window? i.e. if 10 tabs is the point where tab titles loose their usefulness on average screens (maximized window on 800x600), then are there realistic uses for insertion of multiple groupings within a window? 3 groups of 3 tabs seems to be the maximum extent of utility for allowing users to do this, however, are 3 tab groups (or less) going to be common enough that we should provide this feature. Or should we assume that advanced to intermediate types, for whom the feature is primarily intended, will likely have large enough screens and allow them to the flexibility to get themselves into and out of akward circumstances? Perhaps it is more important to honor the bookmark / tab relationship by being consistent.
  • Should Home Page spawn in every new tab? In the first tab of each new window (on a locked Tab Bar)? Similarly, should a proposed "Home Set" spawn in every new window?
  • Sidebar Considerations- How to close the ambiguity between sidebar tabs and page tabs. We have a proposal for this, but is so encompassing it requires a a concurrent redesign of sidebar.

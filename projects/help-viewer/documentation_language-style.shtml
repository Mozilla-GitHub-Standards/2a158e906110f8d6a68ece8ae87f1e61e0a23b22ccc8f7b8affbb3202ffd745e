Language Style

Clarity of writing usually follows clarity of thought. So think what you want to say, then say it as simply as possible. The Economist Style Guide

When a program malfunctions, the end user has no idea what has happened. He may not even realize anything has happened. One action the end user will likely take is to seek help on his situation through websites and Help documentation. Given the crucial role that Help documentation can play in aiding and retaining an inexperienced user, Help documents must be uniform, accurate, and concise. These guidelines describe how to consistently and clearly write Help documentation; this will keep Help documents uniform, accurate, and succinct.

Draft 1.0
Jeff Walden

Purpose

The purpose of any help document is to succinctly explain how something works. Manuals do this through:

  • Clear, structured instruction,
  • Helpful graphics, and
  • Useful information

Naturally, the Help documents of all Mozilla programs strive for these goals. Help documents cannot be useful if they are not consistently and clearly written. By setting a standard writing style, Help documents will become more cohesive and clear. This should keep Help documentation better serving its purpose: instructing end users on how to use mozilla.org programs.

Overall Mozilla Guidelines

There are no explicit Mozilla guidelines to follow. In keeping with app consistency, however, Help documents must conform with the text of the user interface. This means that spellings for terms, term names, and the information covered in the UI must be consistent. While information covered will be a constant issue, term names and spellings are easily corrected. Term standardizations are listed below.

Voice

Help documentation should be enabling, authoritative, and friendly. It should seek to give the user the knowledge to solve his problems. When explaining or enumerating tasks, help documentation should speak of what the user can do or what the program lets the user do. Avoiding the use of passive voice is also nearly always a good idea for similar reasons. It should never sound uncertain (a certain amount of choice is of course acceptable). It should be simply written in a slightly formal yet conversational manner. What this means practically is that direct reference to the user is acceptable and expected, while direct references to the author/speaker shouldn't be used.

When referring to the program, never use a generic term when the product name will do (e.g., "Mozilla Firefox" instead of "the browser"). Additionally, use the product's formal name only in topic introduction and more prominent places. Reinforcing branding ("Mozilla Firefox") is preferable for the most visible locations, while shortness ("Firefox") is preferable if possible in other locations for succinctness. (Please note that usually some method exists for entering the program's name without directly typing it — currently Help documentation uses entities — such as &brandShortName; and &brandFullName; — to encode product names.

Also, to avoid description rot, never mention anything in a time-dependent manner. For example, "new" features will not always remain "new", and "planned" features (hopefully) will not always remain "planned". These things make Help documentation rot, and such references have become bugs at least once already in docs.

Links

Link texts should make sense out of context. In essence this means that if a list of all the links in a page were made, each link would directly connect with the target of the link. In particular, the phrase "click here" must never be used. Giving links relevant text makes them easier to distinguish from the rest of the document and aids interpretation by screen readers and other such programs. Links may have title attributes if desired, but they are not required.

Terms

Some of these terms may be specific to one application. For example, Bookmarks Toolbar is Firefox-specific, so the Mozilla Suite may continue to use Personal Toolbar. In such cases please use the app-preferred term. In all cases the current user interface is the ultimate reference; if you happen to discover a conflict, please let us know immediately. If a term is not in the user interface and you believe it deserves to be standardized, let us know about this as well. If your term does not fit any of these situations, use the more cutting-edge version if it has gained substantial usage.

Any changes and additions to this list will be posted here as they occur.

Mechanics

Perhaps most importantly, always use proper grammar and spelling. While many users will not notice a difference, the more educated users among our target audience whose opinions should be particularly respected will notice when "it's" is incorrectly used instead of "its". Such mistakes are usually trivial, but they tend to easily irritate when present and are unprofessional. If you are at all unsure of the mechanics of your Help contributions, someone can always review your changes and additions to ensure correctness.

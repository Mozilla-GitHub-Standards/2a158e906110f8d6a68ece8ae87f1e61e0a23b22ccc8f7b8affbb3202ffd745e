Localization Note Guide

Version: 1.2
Date: Nov. 5. 1999
Created by: Chang Jung

1, Overview
    This guide line describe that how to make Localization Notes on DTD/Properties files for Seamonkey project.
 

2, Syntax

  • DTD files
    • single line

      • <!-- LOCALIZATION NOTE (<Entity id>): [<keyword>] [<arguments>] -->
       
    • multi-line

      • <!-- LOCALIZATION NOTE (<Entity id>): [<keyword>]
          <argument 1>
          <argument 2>
                  :
          <argument n>
      -->
  • Properties files
    • single line

      • # LOCALIZATION NOTE (<Entity id>): [<keyword>] [<arguments>]
       
    • multi-line

      • # LOCALIZATION NOTE (<Entity id>): [<keyword>]
      #   <argument 1>
      #   <argument 2>
      #            :
      #    <argument n>


3, Keywords

  • General Localization comments
    • Use no keyword
    • the contents can be any helpful explanation or description to translate
      • Accurate meaning

        • <!-- LOCALIZATION NOTE (Menu.label.string): the "Left" means opposite meaning of Right -->
      • Concatenation issue

        • <!-- LOCALIZATION NOTE (Status.label.string1):
               Translatate (Status.label.string1) with (Status.label.string2)
               but make sure the "%s" is belong to the first message.
        -->
      • Guide line

        • <!-- LOCALIZATION NOTE (Status.label.string):
              keep the order of two "%lu" when the grammer need to change the order.
              For example, in Japanese cases, use following message.
                      %s Receiving: message headers %lu (total %lu)
        -->
      • Not to be translated one string: "<string1>"

        • <!-- LOCALIZATION NOTE (Window.title.label): don't translate "Sidebar"  -->
         
      • Not to be translated more than 2 strings: "<string1>", "<string2>" ,..., "<stringn>"

        • <!-- LOCALIZATION NOTE (Window.title.label): don't translate "%s", "&quote;", "ISO"  -->
         
  • Not to be translated
    • Use the keyword DONT_TRANSLATE
    • indicate the whole message should not be translated.
    • Syntax:
        • <!-- LOCALIZATION NOTE (<Entity id>): DONT_TRANSLATE -->
    • Example:
        • <!-- LOCALIZATION NOTE (Window.title.label): DONT_TRANSLATE -->
         
  • File scope comment
    • Use the keyword FILE
    • indicates the whole file need to follow certain direction
    • Syntax:
        • <!-- LOCALIZATION NOTE : FILE <arguments> -->
    • Example:
        • <!-- LOCALIZATION NOTE : FILE
             This file contains a lot of "Accesskey"s.
              Please make sure to read "Translation README" file
              before translating those messages.
        -->


4, Updated History

  • 10/03/1999: V0.1 New Draft created
  • 10/15/1999: V0.2
    • Delete consecutive line indicator: because entities' order could be changed.
  • 11/03/1999: V1.0
    • Change the Syntax for allowing general comments besides of keywords.
    • Delete TRANSLATE_THIS_WAY keyowrd: because it is duplicate indication of "Localization Notes"
    • Delete HOW_TO_PRODUCE keyowrd: because it is more like to "QA Notes"
    • Reduce the scope of DONT_TRANSLATE
      • Use DONT_TRANSLATE keyword when the whole messege need to be untouched.
      • Delete arguments of DONT_TRANSLATE and those could be represented by general L10N notes
    • Add FILE keyword: because some of files need general comment for whole file
  • 11/04/1999: V1.1
    • Change and clean relative position comments like 'below line' or 'following message'
    • Delete FILE keyword and merge this feature to general note
  • 11/05/1999: V1.2
    • Revive FILE keyword: using file name in note has disadvantage of synchronization issue.
      • Whenever the file name is changed, the note need to be updated -> too much work
      • FILE keyword is much easier to implement in tool.
      • using file name and using FILE keyword at the same time is redundant.
5, Note
  • The examples of Keywords are for DTD only,  Properties files can use the same keyword.
  • Accesskey or Command key will not have any Localization note
    • Those entities can be translated in most European locales but not in Double-Byte locales
    • Accesskey/Command Key Traslation guideline need to be described in other documentation
  • Acronyms ("ISO"), HTML identifiers ("&quote;") and variables ("%s" or "%d") does not required to have its own Localization note to indicate explicitly not to be translated. But since they could cause serious problems (HTML identifiers and variables),  we need a validation tool for checking those.
  • Do not use "--" within Localization note. Most of parsers recognize the string as the end of comment, "-->".

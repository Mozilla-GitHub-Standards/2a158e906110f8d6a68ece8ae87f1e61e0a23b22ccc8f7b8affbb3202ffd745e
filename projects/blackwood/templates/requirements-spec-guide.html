Requirements Specification Guide

 

Purpose

The intent of this guide is to make it easy to write a precise and complete specification for a software project.

How to Write a Specification

To write a specification, copy the Requirements Specification Template to a working directory, rename it as you like, then replace all bracketed text by information pertinent to your project. (Contact edburns@acm.org if you have comments or suggestions regarding the template.)

When you have finished, it is a good idea to verify the completeness of your specification by checking it against the list below:

Optional Checklist


The requirements specification should:
  • describe the flow and content of information
  • elaborate all software functions
  • describe all events that affect the system
  • uncover design constraints
  • show evidence and conclusions from research on prior projects that addressed the same problem. Specifically, list problems with each prior project.
  • specify performance constraints
  • serve as the basis for a Preliminary User's Manual
  • describe the design principles that will be followed; for example, design for maintainability
  • describe the benefits of a successful implementation
  • describe the environment in which the software is to run
  • be a cognitive model; that is, it must be described in the user domain with user terms.
  • separate functionality from implementation
  • be useful for validating the correctness of an implementation
  • be tolerent of incompleteness and be augmentable
  • be modular and loosely coupled

Camino Development Information

Mozilla version

Camino is now building from the “trunk” of the Mozilla tree, which means that nightly Camino builds pick up all the changes that other developers are making in the main Mozilla development effort. This has the benefit of keeping Camino current, but the downside that quite often, nightly builds will be unstable.

Camino 0.7 was the last release that was built from the CHIMERA_M1_0_1_BRANCH.

Recent changes

Camino now requires Mac OS X 10.3 (Panther) and XCode to build. We're sorry for any inconvenience and upgrade hassles this may cause.

You can see what’s happening in the Camino “tree” (the “tree” is the source code repository, as managed by the CVS version control system) right now by looking at the Camino tinderbox. This has a column for each machine that is continuously pulling and building the Camino source code, to check that it builds and runs OK.

You can also see what code changes have been made in the last day, and the last week (might be slow to load). Note that you'll see changes that affect Camino-specific code (in mozilla/camino, and the rest of the tree.

To see a summary of the changes made since the last release build, see the release notes.

Bug Tracking

If you find a problem that you think might be a bug in Camino, please don’t jump in and file it right away. Having low-quality and duplicate bugs in the bug system simply wastes the time of developers and QA. Instead, follow these steps.

  1. First, try the most recent nightly build, to see if the problem still exists.
  2. Next, try to reproduce the bug with a recent Mozilla build, to see if the bug happens there. If so, the bug should be filed as a Mozilla bug (only, of course, after searching to see if the bug has already been filed). Bugs related to how pages are laid out are almost always Mozilla bugs, and are highly likely to have been filed already.

    3. Now you can go ahead and search the Camino bugs to see if someone has already reported your problem.

  3. Search the open Camino bugs, but be aware that your terminology might not match that of others (e.g. “URL bar” vs. “Location bar”), so you might not find what you are looking for the first time. Be diligent!
  4. Look at Camino bugs changed in the last day, or the last week.
  5. Look at the list all open Camino bugs to see if your issue has already been filed.

    6. If you didn’t find an existing bug for your problem, and you’re sure that it’s a problem with Camino, then you can go ahead and file the bug.

    If you are new to Bugzilla, please read the bug writing guidelines first. To be able to file and comment on bugs, you’ll need to make yourself an account. Note that accounts can, and will, be revoked if abused.

  6. Report a new Camino bug using the Bugzilla helper

    7. Put as much relevant detail into the bug report as you can. If Camino crashed, you should have a Crash Report (see the FAQ item for how to get this), which you should add to the bug as an Attachment. If you submitted a Talkback incident, give the incident ID (you can copy and paste this from the Talkback window). If you were on a certain page when it crashed, give the URL. Mention whether the page contains plugins. Try to find the simplest way of causing the crash that you can. If it crashes when you are supplying data to Camino (e.g. importing bookmarks), attach the relevant data (in this case, the bookmarks file being imported). The better your bug report, the more chance there is of a developer taking note, and fixing it.

How to build Camino

Camino is fully open source (which means you have access to the source code, to study, build and modify). The source is available from the Mozilla CVS server. To pull and build Camino, see the build instructions. Note that you‘ll need just over 2GB of disk space for a complete debug tree.

 We need a complete understanding of what build time and runtime dependencies embedding has. There are various places where we utilize services internally (a runtime dependency) which we need to isolate andaccommodate. Bad dependencies will need to be broken.

Builds
 We need to create "embedding" builds which reflect our dependencies.  Ideally this is reflected in a build configuration inwhich the builder  can say things like "make embedding editor|browser|foo." Shared libraries  are also expensive to some customers, so we should beable to handle one  large BIN which statically links in all of the otherwisedynamic libraries. There are also issues surrounding exactly *how* (which compile/link flags are used) our clients build mozilla and expectto link w/ our hard link dependency libs (such as xpcom/nspr). There arelinkage incompatibilities.
 build system as well as broad reaching in the code base (for example, automated build configs need consistent module entry points and we are currently not consistent).

De-couple printing from necko
 Currently our printing implementation asks imagelib for images and rebuilds the document w/ the new images (which have higher resolution than the document being displayed). If imagelib misses a cache hit, we ask necko cache which is bad because in offline mode, necko may not have the data. Depending on our solution to external data feeding of layout, we may run into a scenario there in which necko doesn't have the data either.
 printing, necko, imagelib

Editor Support
 Customers need the ability to generate rich text(HTML) content in a text area. They would like two methods of style control: programatic, which would allow them to call an interface to apply a style (color, size, etc) to a text selection, and toolbar support. Ideally the toolbar couldbe either docked (inside the text content area) or floating.
 nsIDocShell, docshell, editor XUL files, nsIWebBrowserChrome, editor and editor embedding apis.

Design of Auditing in the NSS Software Cryptographic Module

<< DRAFT >>

Author: Wan-Teh Chang

Audited Events

crypto_officer_authentication_failure

This event type satisfies the FIPS 140-2 requirement that attempts to provide invalid input for crypto officer functions shall be recorded by the audit mechanism.

invalid_new_crypto_officer_authentication data

This event type satisfies the FIPS 140-2 requirement that attempts to provide invalid input for crypto officer functions shall be recorded by the audit mechanism.

If the authentication mechanism for the cryptographic module is passwords, this event is recorded if the crypto officer attempts to set his initial or new password and the new password is too weak (e.g., too short) or contains invalid characters.

invalid_new_cryptographic_module_user_authentication data

This event type satisfies the FIPS 140-2 requirement that attempts to provide invalid input for crypto officer functions shall be recorded by the audit mechanism.

If the authentication mechanism for the cryptographic module is passwords, this event is recorded if the crypto officer attempts to set the user's initial or new password and the new password is too weak (e.g., too short) or contains invalid characters.

addition_of_user_to_crypto_officer_role

This event type satisfies the FIPS 140-2 requirement that the addition or deletion of an operator to/from a crypto officer role shall be recorded by the audit mechanism.

MR Note: This event type is closely related to the auditable event "modifications to the group of users that are part of a role" prompted by the requirement "Security Roles (FMT_SMR.2)" in MR. FMT_SMR.2.1 requires that the TSF shall maintain a cryptographic administrator role.

libaudit Note: AUDIT_ROLE_ASSIGN

NSS Note: This event type does not apply to the NSS software cryptographic module.

deletion_of_user_from_crypto_officer_role

This event type satisfies the FIPS 140-2 requirement that the addition or deletion of an operator to/from a crypto officer role shall be recorded by the audit mechanism. An alternative is to have an event type that combines this event and the previous event.

MR Note: This event type is closely related to the auditable event "modifications to the group of users that are part of a role" prompted by the requirement "Security Roles (FMT_SMR.2" in MR. FMT_SMR.2.1 requires that the TSF shall maintain a cryptographic administrator role.

libaudit Note: AUDIT_ROLE_REMOVE

NSS Note: This event type does not apply to the NSS software cryptographic module.

Auditable Events

cryptographic_module_self_test_failure

MR Note: this event type is closely related to the auditable event "execution of the cryptography self tests and the results of the tests" prompted by the requirement "TSF Testing (for cryptography) (FPT_TST.1(1))" in MR.

libaudit Note: AUDIT_ANOM_CRYPTO_FAIL

cryptographic_module_audit_data_review

This event type satisfies the FIPS 140-2 requirement that the audit mechanism shall be capable of auditing operations to process audit data stored in the audit trail.

NSS Note: This event type does not apply to the NSS software cryptographic module.

cryptographic_module_audit_data_search

This event type satisfies the FIPS 140-2 requirement that the audit mechanism shall be capable of auditing operations to process audit data stored in the audit trail.

NSS Note: This event type does not apply to the NSS software cryptographic module.

request_to_change_cryptographic_module_authentication_data

This event type satisfies the FIPS 140-2 requirement that the audit mechanism shall be capable of auditing requests to use authentication data management mechanisms.

cryptographic_module_initialization

persistent_key_generation

persistent_key_import

NSS Note: C_UnwrapKey. C_SetAttributeValue(CKA_TOKEN)

persistent_key_export

NSS Note: C_WrapKey. C_GetAttributeValue

persistent_key_deletion

cryptographic_module_authentication_attempt

This event type satisfies the FIPS 140-2 requirement that the audit mechanism shall be capable of auditing the use of an authentication mechanism (e.g., login) associated with the cryptographic module.

MR Note: This event type is related to the auditable event "all use of the authentication mechanism" prompted by the requirement "Timing of Authentication (FIA_UAU.1)" and the auditable event "all re-authentication attempts" prompted by the requirement "Re-authenticating (FIA_UAU.6)".

libaudit Note: add AUDIT_CRYPTO_AUTH or AUDIT_CRYPTO_LOGIN?

NSS Note: This event type is recorded whenever an operator attempts to log into the NSS software cryptographic module.

explicit_request_to_assume_crypto_officer_role

This event type satisfies the FIPS 140-2 requirement that the audit mechanism shall be capable of auditing explicit requests to assume a crypto officer role.The wording of this event ("explicit requests to assume a role") seems to come from Common Criteria. See MR Note below.

MR Note: This event type is related to the auditable event "explicit requests to assume a role" prompted by the requirement "Assuming Roles (FMT_SMR.3)" in MR. FMT_SMR.3.1 says the TSF shall require an explicit request to assume any role.

libaudit Note: AUDIT_USER_ROLE_CHANGE?

NSS Note: This event type is recorded whenever an operator attempts to log into the NSS software cryptographic module. (In NSS the user role and the crypto officer role are combined into one role.)

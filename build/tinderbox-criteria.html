Proposed Platform Criteria for the Main Tinderbox Page

Chris McAfee, Brendan Eich

What platforms belong on the main Tinderbox page? This can be a heated topic of discussion, so we've written this document in an attempt to provide some insight as to how the main set of Tinderbox platforms is determined, and to provide some guidance when considering changes to the set of platforms as Mozilla evolves.

An explanation of where we are now:

  • Purpose. The main purpose of the main page is to keep developers working together as a group. This means, don't break other people's builds and slow them down. The builds on this page should represent groups of people doing work on the main mozilla codebase.
  • Technical reasons. Sometimes there is a technical decision to support a particular compiler or platform that will benefit a major group of developers. (E.g. we had a gcc2.7.2.3 build to keep purify working on Unix.)
  • Simplicity. This page needs to be streamlined, too many builds makes things complicated and slows down the group. Make these builds count. The fewer builds we have, the more-focused the group can be, but we need to cover enough platforms to keep everyone working.
  • Regressions. Tinderbox builds can do testing. If we need a machine to do some special tests (e.g. to prevent memory/speed regressions), the main page is a good front-and-center place to do this.

We will use the following criteria when considering whether or not to add a new platform to this page:

  • OS/Compiler Transitions. Sometimes an OS or compiler transition needs to happen over a longer period of time and it becomes necessary to show both the old and new builds simultaneously. (E.g. win16/win32, MacOS 9/X, gcc versions)
  • Critical mass. We shouldn't slow the entire engineering group down for a platform with one developer, or for a platform that isn't contributing to the main codebase.
  • Number of downloads. This might help establish that a build is becoming more important.
  • Relevance. Does this build make sense? This is meant to catch future scenarios that aren't described by parameters known when this document was written.

If a new platform shows up, we need to make sure of the following:

  • Ownership of new builds. A person or group needs to be responsible for maintaining the Tinderbox build, and to provide a source of help for fixing hard-to-solve, platform-specific bustage.
  • Stage the new build first. The build should be staged on the ports page for a while to make sure it is stable and is reporting correct build information.
Process proposal:

Many people share responsibility for maintaining tinderbox.mozilla.org as an effective tool. The sheriffs, staff, build folks, and platform advocates all have a stake. Decisions about main page platforms should be made by consensus, with staff helping to forge agreement and to break deadlocks, if any arise.

To reach consensus quickly, track status, and implement the decision according to the above criteria, we propose using Bugzilla to give responsibility for each platform candidate to a bug assignee. So to nominate a Tinderbox platform, file a bug against the mozilla.org product, in the Tinderbox Platforms component. The bug's assignee will then work to reach consensus quickly, make a decision according to the above criteria, and implement any necessary changes.

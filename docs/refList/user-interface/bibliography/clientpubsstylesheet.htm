Conventions

User Interface Considerations

This style sheet is a working document and is a supplement to the Tech Pubs
Style Guide; please let me know if anything here is incorrect or
inappropriate.

Conventions

Displaying/loading pages
Don't say bring a page; say display a page, transmit a page from a server, load a page. OK to say bring a page to screen.
Avoid saying transfer without some synonym or hint that this is same as loading or transmitting.

Filenames
Use computer voice font. If you are writing in Frame, use 8pt Courier for code listings and 9pt Courier for elements in text. If you are writing in HTML, use the <CODE> tag for elements in text, and the <PRE> or <TT> tags for code listings. See the Writer's Guidelines for more information.

Fonts

  • Boldface--Avoid using boldface for button names, menu items, panel names, and so on. It's not necessary for clarity. You can use boldface for terms that appear in a glossary.
  • Italics--Use for "words as words" when appropriate. Also use italics for replaceables (for example, Window Items is variable "placeholder" for item in Window menu) and book titles.
  • Courier--Use for code and code listings, filenames, file extensions, pathnames, directory names, URLs, HTML tags, and anything the user must type on the screen. See the Tech Pubs Style Guide for more information.
Folder names
Folder names should appear in documentation the way they appear on the computer. For example, System Folder (folder is capped because it's part of the actual name), Printers folder, Control Panel folder, and so on.

Key names
Use initial cap (Shift key, Return key).

Menu/menu item selection
Don't use the "pipe" convention:
Correct: From the File menu, choose New Web Browser.
Incorrect: Choose the File|New Web Browser menu item.

Menu items
No ellipses after menu item in text.
Correct: From the Insert menu, choose Target.
Incorrect: From the Insert menu, choose Target....

Operating systems
Use simple wording, like "On Windows, the field is
labeled x." Say "on Windows" instead of "on a PC." Use the term operating
system instead of platform.
Macintosh: On the Mac OS...
Unix: On Unix...
List versions in order of release. For example, Windows 3.1, Windows 95, Windows NT, and so on.

Quotation marks (From the Tech Pubs Style Guide)
Use quotation marks around UI elements only if they contain lowercase words.

Click the "Do it for me" button.

Special characters
When referring to a special character, symbol, or punctuation mark, give the character in parens after its name.

Windows/Mac terminology
If there's a simple difference in terms, give the Windows term first. For example, "directory names (folder names)." (If
indeed this terminology is still current in the Windows world...)

Dialog box/panel options and fields
When describing a dialog box or panel, avoid saying "the...field" and "the...button" unless it's necessary for
clarity or consistency. For example, say "click Create New Type" instead of "click the Create New Type button"; "select Verify Documents" instead of "click the Verify Documents radio button."



User Interface Considerations

Interface actions. These are per Apple style for clicking, choosing, selecting, and so on.
  • You click an onscreen item such as a link (you don't click on it). It's OK to say that you click in something large, like a window, if that's clearer.
  • You click a button (you don't press it, choose it, or click on it).
  • You click or select a radio button.
  • You click to select or deselect a checkbox (you don't check or uncheck it). If appropriate, you can describe an unselected (for example, empty) checkbox.
  • You choose a menu item.
Interface items. Here are a few names:
  • The menu bar is the series of menu names across the top of the screen.
  • A menu (not a menu bar item) is in the menu bar.
  • A menu item (or menu command) is in a menu
Steps in User Actions (From the Tech Pubs Style Guide)
When describing user actions, present steps in a logical order that corresponds to the sequence the user follows.

Yes:
From the XXX menu, choose YYY.
From the File menu, choose New Document and then Blank

No:
Choose YYY from the XXX menu.
Choose Blank from the New Document submenu of the File menu.

The latter forces the reader to process the sentence backward to arrive at the correct sequence.

Miscellaneous

  • Avoid naming a dialog box unless (or until) necessary.
  • In a dialog box, avoid over-identifying the type of control. For example, don't say "choose one of the Always and Sometimes radio buttons to..." if you can say "Click Always or Sometimes to..." I believe this gives users the instrux they need to perform the task while looking at the interface, but doesn't overload them by describing aspects of the interface.

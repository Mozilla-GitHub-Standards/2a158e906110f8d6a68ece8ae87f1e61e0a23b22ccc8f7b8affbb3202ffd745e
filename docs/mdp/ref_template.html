There are lots of ways to format the top of a page, but in this particular reference example, each object has its own page, and is "spelled" in a table. This is one page in a large set, in other words. These pages may be linked with a frameset or with a link list somewhere higher up.

The table lists the members (in this case "attributes") in the left column, and provides some sketch of the object hierarchy in the right column. By the way, I am using the term "members" here to mean, in object-oriented parlance, all the methods, properties, data, events, and whatever else that are accessible via a particular object's interface.

Anyway, in this right column, only the children of the object are given, but you could just as easily list the base classes of that object, if this were some kind of OO Reference, or the classes that inherit from this object, which is sort of what children are in this example.

Note that in the table above, the attributes that are italicized are attributes of a class this object inherits from. These attributes are actually defined elsewhere in the reference documentation, and the italics provide a visual clue to this.

Each item in the left column of the table is described on this page. The links there are to targets lower in the page. The page is laid out with a big table. The heading for each of the attributes or members is colored to separate it from the others. As in the regular documentation template, you may want to show code listings or other examples in this first general area. A handy way to format these examples follows: 

<box id="parent_box" 
  align="vertical" 
  style="background-color:lightblue;">
    <box flex="1">
       <titledbutton class="borderless" value="Start" />
       <titledbutton class="borderless" value="Stop" />
    </box>
    <spring flex="3" />
</box>

You may not have any description in this top part of the object page. In some cases, the top part is simply a listing of the members further down, in which case the first member appears a few blank lines below the member table.
 

 
align
Description
align is an optional attribute that specifies whether the widget's children are to be aligned vertically or horizontally inside that widget.
Syntax
<element align="vertical | horizontal">
Example
  
 
<menubar align="horizontal" 
  style="background-color:lightblue;">
<menubar align="vertical" 
  style="background-color:lightblue;">
Notes
This is the canonical set of descriptors for a member: Description, Syntax, Example, Notes. It would perhaps be nice to have a separate "Values" descriptor, which would list the possible values for that member or at least the data type. For example, under values for the "align" attribute, you could say simply, "horizontal or vertical," and for attributes that take strings, you could say something like, "a string value representing the name of the element you wnat to include" or something like that.

In the Syntax section, things that aren't literal are italicized. In align's Syntax, for example, horizontal and vertical are not italicized because those are the actual two possibilities. In flex's Syntax, however, "integer" and "percentage" are italicized because some integer or some percentage representing the degree of flexibility is indicated.

Note also that in the description, the attribute itself is bold, and some indication is given about whether the attribute is required or optional. It might actually be better to note this in some more explicit way, actually. Hmmm...at least it's in the top level descriptor.

 
 
 
flex
Description
flex is used to position widgets and establish their flexibility relative to one another.


Syntax

<element flex="integer | percentage">
Examples
  
<box align="vertical" style="background-color:lightblue;">
    <box flex="1">
       <titledbutton class="borderless" value="Start" />
       <titledbutton class="borderless" value="Stop" />
    </box>
    <spring flex="3" />
</box>
  
<box align="vertical" style="background-color:lightblue;">
    <box flex="100%">
       <titledbutton class="borderless" value="Start" />
       <titledbutton class="borderless" value="Stop" />
    </box>
    <spring flex="50%" />
</box>
Notes
Flex is one of the principal ways to size and arrange widgets in the UI. Using boxes, springs, flex, and the functional XUL widgets you need, you can layout the interface in any way you choose.

In the first example above, the inner box widget is vying for space with the spring widget. Both of them have been given flex, but the greater flex of the spring widget has resulted in a spring element (below the two buttons) that is three times as tall as the buttons themselves. 

A few more things about the layout of the topic in the HTML: I have tried to use regular-sized headers (e.g.,Description, Syntax, et al) just because it looks better, but you could also use some actual heading, as long as the space around that heading is not too large. In all cases the actual content, including the example tables, are set off with the <blockquote> tag which indents it and keeps you from having to use any space characters or other weirdness.

When flexible widgets (widgets for which some flex value has been specified) are arranged with other, unflexible widgets, the flex of the flexible widgets is always understood to be complete. The inflexible widgets in this case take up only as much space as they need to fulfill their other requirements: that their values be displayed, that the style considerations like padding and borders are met, and so on.
